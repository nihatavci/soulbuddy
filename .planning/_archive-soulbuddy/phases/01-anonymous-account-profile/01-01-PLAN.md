  <files>supabase/migrations/</files>

  <action>
    Create the migration file via `supabase migration new anonymous_profiles` (generates a
    timestamped `supabase/migrations/<ts>_anonymous_profiles.sql`). Define:
    (a) Table `public.profiles`: `id uuid primary key default gen_random_uuid()`,
    `user_id uuid not null unique references auth.users(id) on delete cascade`,
    `age integer check (age >= 18 and age <= 120)`,
    `gender text check (gender in ('male','female','nonbinary','other','undisclosed'))`,
    `location text` (free-text city/region ASL signal — NOT precise coordinates),
    `onboarded boolean not null default false`,
    `created_at timestamptz not null default now()`,
    `updated_at timestamptz not null default now()`.
    DELIBERATELY OMIT any `display_name`, real-name, `email`, or `avatar_url` column.
    (b) `alter table public.profiles enable row level security;`
    (c) Owner-only policies keyed on `auth.uid() = user_id`: `profiles_select_own` (for select),
    `profiles_insert_own` (with check), `profiles_update_own` (using + with check). No delete policy needed.
    (d) Column-limited view `public.public_profiles` selecting ONLY `user_id, age, gender, location`
    from `profiles` (default security-definer semantics so all users can read ASL of everyone).
    Do NOT join `auth.users`; do NOT expose any timestamp/email/name column.
    `grant select on public.public_profiles to anon, authenticated;`
    (e) `create index idx_profiles_user_id on public.profiles(user_id);`
  </action>
  <verify>
    <automated>f=$(ls supabase/migrations/*_anonymous_profiles.sql); grep -q "auth.uid() = user_id" "$f" && grep -q "create view public.public_profiles\|create or replace view public.public_profiles" "$f" && ! grep -iqE "display_name|avatar_url|(^|[^a-z])email([^a-z]|$)" "$f"</automated>
  </verify>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user A ↔ user B (cross-user reads) | Untrusted: one user must never read another's real identity |
| client ↔ Postgres RLS | Client queries are untrusted; RLS + view are the enforcement point |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Information Disclosure | profiles base table | mitigate | Owner-only RLS (`auth.uid() = user_id`) on select/insert/update; cross-user base reads return 0 rows |
| T-01-02 | Information Disclosure | public_profiles view | mitigate | View exposes ONLY user_id, age, gender, location; no email/name/avatar column, no `auth.users` join |
| T-01-03 | Information Disclosure | profiles schema design | mitigate | Table deliberately has no real-name/email/avatar column — identity leakage impossible by construction |
| T-01-04 | Elevation of Privilege | RLS insert/update | mitigate | `with check (auth.uid() = user_id)` prevents writing rows attributed to another user |
| T-01-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed this plan (all deps already in package.json); no supply-chain surface added |
</threat_model>
