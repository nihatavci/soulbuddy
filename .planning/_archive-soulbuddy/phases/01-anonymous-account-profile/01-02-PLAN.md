<objective>
Make the app boot and run locally without any external SDK keys, make email
sign-up/login GENUINELY REACHABLE through real navigation, and verify the full
email auth slice against the LOCAL Supabase stack from Plan 01: sign up, log in,
session persistence across restart, and log out.

Purpose: AUTH-01/02/03. The AuthContext + encrypted-MMKV persistence already exist,
and `(auth)/sign-in.tsx` / `(auth)/sign-up.tsx` already call `supabase.auth.*`
directly. BUT the current AppGate (`app/(app)/index.tsx`) auto-creates an anonymous
session and gates on local MMKV flags, so a fresh user NEVER reaches the auth
screens. This plan fixes reachability by rerouting unauthenticated users to
`(auth)/sign-in` before the anonymous session fires, guards the excluded SDKs so
startup does not crash, points the client at local Supabase, and verifies the four
auth behaviors end-to-end.
Output: a locally runnable app where a fresh user can complete email sign-up and
login through the UI, with persistent sessions and working logout.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client ↔ Supabase Auth | Credentials + session tokens cross here; tokens stored on device |
| user A ↔ user B | A user's auth email (real identity) must never be reachable by another user |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-05 | Information Disclosure | auth.users email | mitigate | Email lives only in `auth.users`; never joined into `profiles`/`public_profiles`; not selected by any app query (identity-leakage invariant preserved) |
| T-01-06 | Information Disclosure | session token at rest | mitigate | Reuse existing encrypted-MMKV adapter (key in expo-secure-store / Keychain) — no plaintext token storage |
| T-01-07 | Spoofing | sign-in/sign-up | mitigate | Auth handled by Supabase Auth (password hashing server-side); client only forwards credentials |
| T-01-08 | Repudiation | log-out | mitigate | `signOut` clears Supabase session + query cache; AppGate then redirects to `(auth)/sign-in` |
| T-01-13 | Information Disclosure | anonymous auto-session | mitigate | Removing `signInAnonymously()` from AppGate ensures users authenticate with a real identity before any profile write — no ambiguous anonymous rows |
| T-01-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed; no supply-chain surface added |
</threat_model>
